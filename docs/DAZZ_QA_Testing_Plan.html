App 1: DAZZ Producciones

Auth Module

FLOW: Login

Preconditions: User has registered account. App at /login.
Steps: Enter username/email + password, click Login.
  • Outcome 1: Valid credentials → tokens stored, redirect to /dashboard
  • Outcome 2: Invalid credentials → "Incorrect username/email or password" (401)
  • Outcome 3: Account locked (5+ failed attempts) → "Account temporarily locked" (429)
  • Outcome 4: Lock expired → counter resets, normal login proceeds
  • Outcome 5: Empty fields → browser native validation blocks submit
  • Outcome 6: Server error → generic error shown

FLOW: Token Refresh

Preconditions: User authenticated, access token expired.
Steps: Automatic — triggered by 401 response on any API call.
  • Outcome 1: Valid refresh token → new access token, request retried transparently
  • Outcome 2: Invalid/expired refresh token → clear storage, redirect to /login
  • Outcome 3: No refresh token in storage → immediate redirect to /login
  • Outcome 4: Concurrent 401s → queued, all resolved when refresh completes
  • Outcome 5: Refresh endpoint down → all queued requests fail, logout

FLOW: Set Password (from email link)

Preconditions: User received set-password email with valid token.
Steps: Click email link, enter new password, submit.
  • Outcome 1: Valid token + valid password → password set, tokens revoked, success message
  • Outcome 2: Expired token → "Token inválido, expirado o ya usado" (400)
  • Outcome 3: Already used token → same error (400)
  • Outcome 4: Password too short (<8) → Pydantic 422
  • Outcome 5: Password without uppercase → 422
  • Outcome 6: Password without number → 422
  • Outcome 7: Password without special char → 422
  • Outcome 8: User deleted between token creation and use → "Usuario no encontrado" (404)

FLOW: Forgot Password

Preconditions: User at /login, clicks "Forgot password".
Steps: Enter email, submit.
  • Outcome 1: Email exists → reset email sent, generic message shown (no leak)
  • Outcome 2: Email doesn't exist → same generic message (security)
  • Outcome 3: Rate limited (>3/hour) → 429
  • Outcome 4: Email send fails → silent failure, generic message shown

FLOW: Logout

Preconditions: User authenticated.
Steps: Click logout button.
  • Outcome 1: Valid refresh token → revoked server-side, storage cleared
  • Outcome 2: Invalid/already revoked token → error caught silently, storage cleared anyway
  • Outcome 3: Network error → storage cleared anyway (catches all)

Projects Module

FLOW: Create Project

Preconditions: User authenticated with at least one company assigned.
Steps: Navigate to /projects/create, fill name, select company, select responsible, submit.
  • Outcome 1: All fields valid → project created, redirect to /projects/:id
  • Outcome 2: Company not found → 404
  • Outcome 3: No permission for company → 403
  • Outcome 4: Responsible not in company → 400 "El responsable X no pertenece a la empresa"
  • Outcome 5: No companies assigned to user → red warning, cannot submit
  • Outcome 6: Single company → auto-selected
  • Outcome 7: OC matches existing OC_PENDING invoices → auto-linked on creation

FLOW: View Project

Preconditions: Project exists in database.
Steps: Navigate to /projects/:id.
  • Outcome 1: Authorized → project + tickets displayed
  • Outcome 2: Not authorized → 403 "No tienes permiso"
  • Outcome 3: Project not found → 404, redirect to dashboard
  • Outcome 4: Closed project → upload disabled, reopen button shown
  • Outcome 5: No tickets → empty state with upload hint
  • Outcome 6: ADMIN/BOSS/owner → delete button visible
  • Outcome 7: WORKER not owner → no delete button

FLOW: Upload Tickets (multi-file)

Preconditions: Project open, user authorized.
Steps: Select or drag files, click upload.
  • Outcome 1: Valid image <10MB → IA extracts data → ticket created
  • Outcome 2: Image >3MB → auto-compressed to 3MB before upload
  • Outcome 3: Image >10MB → rejected by backend
  • Outcome 4: PDF file → accepted, pages extracted
  • Outcome 5: Duplicate file (same SHA-256) → 409 "duplicate_hash" (amber card, NOT retried)
  • Outcome 6: Duplicate invoice number (different file) → warning only, ticket created
  • Outcome 7: IA confidence <0.5 → warning in notes
  • Outcome 8: Math incoherence (base+IVA-IRPF != total, >0.02EUR) → warning in notes
  • Outcome 9: Missing required fields (provider/date/total) → warning in notes
  • Outcome 10: Foreign invoice → currency detected, exchange rate fetched, converted to EUR
  • Outcome 11: Claude AI error → ticket created with "Error en extracción", total=0
  • Outcome 12: Forbidden extension → 400 "Extensión no permitida"
  • Outcome 13: Dangerous filename (path traversal) → 400
  • Outcome 14: Multiple files → sequential upload, individual results per file
  • Outcome 15: Partial success → retry button for failed files only (excludes duplicates)
  • Outcome 16: No files selected → toast warning
  • Outcome 17: Exchange rate API failure → graceful fallback (rate=1.0 or error in notes)
  • Outcome 18: Cloudinary upload fails → 500

FLOW: Review/Edit Ticket

Preconditions: Ticket exists, user has access to parent project.
Steps: Navigate to /tickets/:id, edit fields, save.
  • Outcome 1: Edit fields + save → marked as reviewed, redirect back
  • Outcome 2: Change base_amount → recalculates IVA, total automatically
  • Outcome 3: Delete ticket → removes from project, updates totals, cleans Cloudinary
  • Outcome 4: Foreign ticket → international section shown, currency symbol in amounts
  • Outcome 5: Duplicate invoice_number warning → amber banner with link to other ticket
  • Outcome 6: Navigate between tickets → prev/next arrows, URL changes
  • Outcome 7: Statistics context (?filter=international) → back button goes to /statistics
  • Outcome 8: Custom payment status → free-text input mode
  • Outcome 9: Unsaved changes + navigate away → beforeunload warning
  • Outcome 10: Error ticket (AI failed) now corrected → updates project totals correctly

FLOW: Close Project

Preconditions: Project has at least one ticket, user authorized.
Steps: Click Close, review preview, add recipients, confirm.
  • Outcome 1: Valid close → Excel generated, downloaded, email sent, redirect to dashboard
  • Outcome 2: Email send fails → warning toast with reason, Excel still downloaded
  • Outcome 3: No tickets → cannot close (button disabled)
  • Outcome 4: No email recipients → warning toast, blocks
  • Outcome 5: Project already closed (race condition) → 409 "Project is already closed"
  • Outcome 6: Double close prevention → SELECT FOR UPDATE ensures atomicity

FLOW: Reopen Project

Preconditions: Project is closed, user authorized.
Steps: Click Reopen button on closed project view.
  • Outcome 1: Success → status=en_curso, closed_at=null
  • Outcome 2: Not authorized → 403

FLOW: Delete Project

Preconditions: User is ADMIN, BOSS of company, or project owner.
Steps: Click delete, confirm in dialog.
  • Outcome 1: Success → all tickets deleted (Cloudinary cleanup), project deleted, redirect
  • Outcome 2: Not authorized → 403
  • Outcome 3: Cloudinary cleanup fails → logged, project still deleted

F-001
Missing 404 Route in Main App
DAZZ Producciones has no catch-all route in the React Router configuration. Navigating to any undefined path (e.g., /nonexistent) renders a blank page with no feedback. A wildcard route redirecting to /dashboard or showing a "Page Not Found" component should be added.
MEDIUM
F-002
Login Page Does Not Redirect Authenticated Users
A user who is already logged in and visits /login sees the login form instead of being redirected to /dashboard. This creates confusion. The Login page should check AuthContext and redirect if tokens are present.
LOW
F-003
Users Page Missing Loading State on Action Buttons
The Create, Edit, and Delete buttons on the Users management page do not show loading states during API calls. This allows double-clicking to trigger duplicate requests, potentially creating duplicate users or sending double delete requests.
MEDIUM
F-004
Drop Zone Accepts Any File Type on Drag
The ticket upload drop zone visually indicates only JPG/PNG/PDF are accepted, but the drag-and-drop handler does not filter by file type on the client side. Non-supported files are sent to the backend which rejects them, but the UX would be better with client-side validation on drop.
LOW
F-005
EditData Page Silently Swallows Non-401 Errors
In the Supplier Portal, the EditData page catches all errors but only handles 401 (redirect to login). For any other error (400, 500, network), the user receives no feedback. A toast notification should be shown for non-401 errors.
HIGH
F-006
Notifications Mark-Read Has No Error Handling
In the Supplier Portal, the mark-as-read function uses an optimistic update but has no .catch() handler. If the API call fails, the UI shows the notification as read but it remains unread server-side. On next reload it reappears as unread, confusing the user.
MEDIUM
F-007
Dashboard and Layout API Failures Are Silent
When the Admin Proveedores dashboard or Layout component fails to load data from the API, no error indication is shown to the user. KPIs display 0 and feeds appear empty, which is indistinguishable from "no data" state. A connection error banner would improve clarity.
MEDIUM
F-008
ProjectCloseReview Still Uses window.confirm
The project close confirmation uses the browser's native window.confirm() dialog, which is inconsistent with the ConfirmDialog component used elsewhere in the app. This breaks the visual consistency and dark theme experience.
LOW
F-009
beforeunload in ReviewTicket May False-Positive
The unsaved changes detection in ReviewTicket uses JSON.stringify to compare current state vs. original. Floating-point arithmetic can cause false positives (e.g., 10.1 + 10.2 = 20.299999... vs 20.3) triggering the "unsaved changes" warning even when data is unchanged.
MEDIUM
F-010
No Rate Limiting Feedback on Login in Frontend
The backend correctly implements a 5-attempt lockout, but the frontend does not show a countdown or remaining attempts indicator. Users see the generic "Account temporarily locked" message with no guidance on when they can retry.
LOW
F-011
Supplier Upload Block References Email Instead of Self-Service
When a supplier without an IBAN tries to upload an invoice, the blocking message references contacting via email. Since the portal has a self-service IBAN change feature in the Profile section, the message should direct users there instead.
LOW
F-012
InvoicesList Load Failure Is Completely Silent
When the supplier portal's invoice list fails to load (API error, network issue), no error toast or message is shown. The user sees an empty list identical to "no invoices" state. This can mislead suppliers into thinking they have no invoices.
HIGH
F-013
OCSelector Shows Empty List When All Prefixes Inactive
In the OCSelector component, if all OC prefixes for a company are inactive, step 2 of the OC creation flow shows an empty list with no guidance. The user is stuck with no way to proceed or understand why no options are available.
MEDIUM
F-014
Statistics PDF Export Limited to International Data Only
The PDF export in Statistics only generates the international data section. There is no full-report export option covering all KPIs, quarterly breakdowns, and country data. Users who want a complete statistics report must take screenshots or manually compile data.
MEDIUM
F-015
Concurrent Autoinvoice Generation Not Blocked in UI
While the backend uses SELECT FOR UPDATE to prevent duplicate autoinvoice numbers, the frontend does not disable the Generate button during the API request. A fast double-click could submit two requests. The backend will reject the second, but the error handling creates an unnecessary user experience issue.
HIGH
